# FE 우선 진행 선택 배경과 근거

목표: 본 문서는 왜 프론트엔드(FE)를 먼저 진행하는지, 그로 인한 이점과 전제 조건, 리스크 관리 방안을 명확히 합니다.

---

## 1) 결론
- **전략**: FE 우선 + API 계약 고정 + API 스텁을 활용한 병행 개발(하이브리드)
- **이유**: 사용자 경험/흐름을 먼저 검증하고, 계약에 맞춰 BE를 교체/확장하는 방식이 전체 수정 비용을 최소화하고 속도를 높입니다.

---

## 2) 근거
- **사용자 가치 검증의 우선성**
  - 화면 정보 구조(IA), 상호작용, 접근성(a11y)이 실제 문제를 해결하는지 빠르게 확인 가능
  - 프로토타입 수준에서 유저 피드백 수집 → 큰 방향 수정이 빠르고 저렴함
- **병목 최소화**
  - BE 준비(스키마, RLS, 인프라)에 시간이 걸려도 FE는 목데이터/스텁으로 진척 가능
  - 초기에는 목, 이후 동일 계약으로 실 API로 교체 → FE 변경량 최소화
- **품질 향상**
  - SSR/CSR·테마·반응형·a11y 같은 FE 품질 이슈를 초기에 해결
  - 로딩/에러/토스트 패턴을 먼저 정하면 BE 오류 케이스 대응이 유연해짐

---

## 3) 전제 조건(성공 요건)
- **불변 API 계약**
  - 요청/응답 스키마, 에러 포맷을 문서로 고정(fe-spec.md, be-spec.md 참조)
  - Zod 스키마로 타입 확정 → FE/BE 동일 참조
- **서비스 레이어 단일화**
  - FE는 서비스 훅/클라이언트만 의존 → 내부 구현을 목→실서비스로 교체
- **테스트 가능 목/스텁**
  - /app/api/* 경로에 스텁 구현(계약에 맞춘 페이로드/에러), SWR로 연결

---

## 4) 리스크와 대응
- **계약 변경 리스크**: 스펙 변경 시 영향 범위 파악 어렵다
  - 대응: 변경 관리 표준화(CHANGELOG), 공통 타입/스키마 한 곳에서만 수정
- **목/실서비스 차이**: 데이터 스케일/성능 이슈를 늦게 알 수 있음
  - 대응: 조기에 페이징·인덱스·필터 계약 포함, 샘플 대량 데이터로 예행연습
- **보안/권한 누락**: FE만 빠르게 나가면 보안이 뒤처질 수 있음
  - 대응: 보안 베이스라인(헤더/CSP/인증 요구사항)을 FE 초반부터 반영

---

## 5) 실행 순서(요약)
1) API 계약 확정(완료) — fe-spec.md, be-spec.md
2) FE 템플릿/레이아웃/상태·이벤트 골격 → SWR + 스텁 연결
3) /api/* 스텁 생성(계약 준수, 에러 케이스 포함)
4) BE(Supabase) 스키마/RLS/인증 구현 후 스텁 치환
5) 접근성/보안/관측성 보강 및 배포

---

## 6) 기대 효과
- 초기 UX를 빠르게 검증 → 불필요한 BE 비용 절감
- 팀 병행 개발 극대화 → 사이클 단축
- 계약 기반으로 리팩토링/확장 용이 → 유지보수성 향상
